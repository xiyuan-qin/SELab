牛马直聘系统

动态建模图表（实验八 · SRS 配套建模产物）

对应 SRS V1.0 附录 C(b) 配套建模产物 · 关联功能需求 FR-J-* / FR-E-* / FR-A-*

概述

本文档补充 SRS V1.0 中尚未给出的动态建模内容，覆盖以下四类动态建模工具：UML 状态图（描述对象生命周期）、Petri 网（刻画并发、同步、互斥与有限资源）、数据流图 DFD（呈现系统数据流向）、OCL 约束（在类图基础上以形式化语言精确描述业务规则）。各图均与 SRS 第 3 章功能需求条目（FR-J-* / FR-E-* / FR-A-*）建立可追溯关系。

1. UML 状态图

建模对象生命周期内的状态变迁规则，是黑盒行为视图。

1.1 投递记录（Application）状态图

覆盖 FR-J-5（简历投递与状态跟踪）、 FR-E-3（简历筛选与管理）、 FR-E-4（面试邀约）。 描述一次投递从创建到归档的完整生命周期，已录用/已拒绝/已撤回/已过期为终态。

关键状态不变式（OCL 见第 4 节）：已撤回、已录用、不合适、已过期为终态，从这些状态不可再迁出；同一投递在 7 天内不允许重新创建（FR-J-5 验收标准）。

1.2 企业账号（Enterprise）状态图

覆盖 FR-E-1（企业注册与资质认证）、 FR-A-1（企业资质审核）、 FR-A-3（用户管理）。

业务规则：仅当企业处于「已认证」状态时方可调用 postJob() 发布职位（详见第 4 节 OCL 前置条件）；管理员的所有状态变更操作必须写入 AuditLog（FR-A-1 验收标准）。

2. Petri 网

用于精确刻画并发、同步、互斥与有限资源场景。符号约定参考 Bobbio (1990) System Modelling with Petri Nets。

符号约定：
  • ○ 库所（Place）：圆形，表示系统的某种条件或状态
  • ■ 变迁（Transition）：实心矩形（瞬时变迁）/ 竖线（带时延变迁），表示事件或活动
  • ● 托肯（Token）：库所内的黑点，表示资源数量或条件成立次数
  • → 普通弧（Arc）：箭头连接库所与变迁
  • ─○ 抑制弧（Inhibitor Arc）：圆头线，目标库所有 token 时禁止变迁触发

2.1 面试邀约与执行流程（含面试间共享资源）

覆盖 FR-E-3（简历筛选与管理）、 FR-E-4（面试邀约）。 本网络综合体现：并发（HR 准备 vs 候选人响应同时进行）、同步（候选人接受 + 面试间空闲 → 才能开始面试）、有限资源（面试间初始 2 个 token，模拟 2 间会议室）、互斥（同一时刻仅 2 场面试并行）。

关键变迁分析

  • t5（开始面试）：同步变迁，需要 p4（候选人接受）和 p6（面试间空闲）同时有 token 才能触发，触发后从二者各取走一个 token。
  • t6（完成面试）：触发后向 p6 回填一个 token（释放面试间），同时向 p8 投入一个 token（归档）。
  • 抑制弧（p5 → t1）：体现"14 天内同一求职者同一职位最多发起 3 次邀约"的业务约束（FR-E-4 验收）。

Petri 网性质

  • 有界性（Boundedness）：p6 的 token 数始终 ≤ 2，系统是 2-bounded；其余库所是 safe（1-bounded）。
  • 守恒性：{p6, p7} 构成 place-invariant，token 总和恒为 2，对应面试间总数。
  • 活性：除终态 p5、p8 外，所有变迁均为 live transition，系统不会死锁。

3. 数据流图（DFD）

以分层方式自顶向下描述系统的数据流动、加工与存储。

3.1 Level-0 上下文图

将整个人才招聘系统抽象为单一加工，描述系统与所有外部实体之间的数据交互。

3.2 Level-1 数据流图

将顶层"人才招聘系统"加工进一步分解为 5 个核心子加工，并显式标出数据存储。

说明：每一加工对应 SRS 中的功能模块——1.0 → FR-J-1/2 + FR-E-1；2.0 → FR-J-3/4 + FR-E-2；3.0 → FR-J-5 + FR-E-3/4；4.0 → FR-A-1/2/3；5.0 → FR-J-6 + FR-E-7。

4. OCL 约束

以对象约束语言（OCL 2.4）补充类图无法表达的业务规则——不变式（invariant）、前置条件（pre）、后置条件（post）。

本节基于 系统建模.html 第 3 节的 UML 类图，为核心类追加约束。

4.1 User 类约束

context User
  inv UsernameNotEmpty:
    self.username <> null and self.username.size() > 0

  inv PasswordStrongEnough:
    self.password.size() >= 8
    and self.password.matches('.*[A-Z].*')
    and self.password.matches('.*[a-z].*')
    and self.password.matches('.*[0-9].*')

  inv RoleValid:
    self.role->includes('candidate') xor
    self.role->includes('enterprise') xor
    self.role->includes('admin')

context User::login() : Boolean
  pre  LoginNotFrozen:    self.failedCount < 5 or self.frozenUntil < now()
  post TokenIssuedIfTrue: result = true implies self.lastLoginAt = now()

4.2 Enterprise 类约束

context Enterprise
  inv UniqueCreditCode:
    Enterprise.allInstances()
      ->forAll(e1, e2 | e1 <> e2 implies e1.creditCode <> e2.creditCode)

  inv ActiveJobsCap:
    self.jobs->select(j | j.status = 'active')->size() <= 50

context Enterprise::postJob() : JobPosting
  pre IsVerified:
    self.isVerified = true
  pre WithinCap:
    self.jobs->select(j | j.status = 'active')->size() < 50
  post JobCreated:
    result.oclIsNew()
    and result.status = 'pending_review'
    and result.enterprise = self

4.3 Application 类约束（重点）

context Application
  inv StatusInDomain:
    Set{'submitted','viewed','invited','interviewing','offered',
        'rejected','withdrawn','expired'}->includes(self.status)

  inv NoDuplicateWithinWeek:
    Application.allInstances()
      ->select(a |
          a <> self
          and a.candidate = self.candidate
          and a.job = self.job
          and a.appliedAt > self.appliedAt - 7.days
          and a.status <> 'withdrawn')
      ->isEmpty()

  inv SnapshotImmutable:
    self.status <> 'submitted'
      implies self.resumeSnapshot = self.resumeSnapshot@pre

context Candidate::applyForJob(job : JobPosting) : Application
  pre  JobIsActive:        job.status = 'active'
  pre  ResumeCompleted:    self.resumeCompleteness() >= 80
  pre  NotRecentlyApplied: Application.allInstances()
                             ->select(a | a.candidate = self
                                          and a.job = job
                                          and a.appliedAt > now() - 7.days
                                          and a.status <> 'withdrawn')
                             ->isEmpty()
  post ApplicationCreated: result.oclIsNew()
                           and result.status = 'submitted'
                           and result.appliedAt = now()
  post NotificationSent:   Message.allInstances()
                             ->exists(m | m.receiver = job.enterprise
                                          and m.kind = 'new_application'
                                          and m.relatedTo = result)

4.4 JobPosting 类约束

context JobPosting
  inv SalaryRangeValid:
    self.salaryMin > 0 and self.salaryMax >= self.salaryMin

  inv DescriptionLength:
    self.description.size() > 0 and self.description.size() <= 5000

  inv OnlyVerifiedEnterprisePosts:
    self.enterprise.isVerified = true

  inv ExpireDateInFuture:
    self.status = 'active' implies self.expireAt > now()

context JobPosting::updateStatus(newStatus : String) : Boolean
  pre  ValidTransition:
    Set{
      Tuple{from = 'draft',          to = 'pending_review'},
      Tuple{from = 'pending_review', to = 'active'},
      Tuple{from = 'pending_review', to = 'rejected'},
      Tuple{from = 'active',         to = 'offline'},
      Tuple{from = 'offline',        to = 'active'}
    }->includes(Tuple{from = self.status, to = newStatus})
  post StatusUpdated: self.status = newStatus
OCL 与 SRS 验收标准对应：OCL 约束是 SRS 各功能需求"验收标准（Fit Criterion）"的形式化版本，可直接转化为后端 Service 层的断言或数据库约束。例如 NoDuplicateWithinWeek 对应 FR-J-5 的"7 天内不可重复投递"；ActiveJobsCap 对应 FR-E-2 的"单家企业有效在岗职位数 ≤ 50"。
